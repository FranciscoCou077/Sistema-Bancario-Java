	\section{Introducción}
	El presente documento detalla la arquitectura algorítmica, las decisiones de diseño y el flujo de desarrollo del proyecto ``Sistema Bancario''. Este sistema fue codificado íntegramente en Java, aplicando los paradigmas de la Programación Orientada a Objetos (POO) \cite{oracle_oop} y haciendo uso intensivo de la librería \textit{Java Collections Framework} \cite{oracle_collections} para
	 la gestión transaccional en memoria.
	
	El objetivo fundamental de este proyecto fue modelar las entidades de un entorno financiero real, construyendo un motor de transacciones (Core Banking) capaz de gestionar identidades de clientes, cuentas de captación (débito), instrumentos de rendimiento (pagarés de inversión) y líneas de crédito. 
	
	\subsection{Flujo de Trabajo y Metodología de Desarrollo}
	El desarrollo del sistema siguió un enfoque iterativo e incremental, dividido en tres fases principales:
	\begin{enumerate}
		\item \textbf{Fase de Diseño Conceptual (V1):} Se diagramó la arquitectura base mediante UML \cite{drawio_uml} y se seleccionaron teóricamente las estructuras de datos (Maps, Sets y Lists).
		\item \textbf{Fase de Codificación y Pruebas Unitarias:} Se construyeron las entidades. En esta etapa, el equipo evaluó la posibilidad de integrar una Interfaz Gráfica de Usuario (GUI) utilizando Java Swing o JavaFX. Sin embargo, debido a las restricciones de tiempo del calendario académico y para priorizar la robustez de la lógica de negocio subyacente, la GUI fue descartada de esta iteración. En su lugar, el equipo pivotó hacia la construcción de una Interfaz de Línea de Comandos (CLI) interactiva altamente validada y libre de errores de ejecución.
		\item \textbf{Fase de Refactorización y Pruebas de Integración (V2 - Final):} Se conectaron los módulos, detectando vulnerabilidades de seguridad financiera que obligaron a reescribir la lógica de validación de fondos, resultando en el diagrama UML definitivo presentado en este informe.
	\end{enumerate}

	\section{Arquitectura del Modelo de Datos y Lógica Bancaria}
	
	\subsection{Desacoplamiento de Identidad y Activos Financieros}
	Una de las características más fuertes de la ingeniería de este proyecto es la separación estricta entre el ID del Cliente y los números de cuenta. En sistemas novatos, es común que un usuario sea directamente "una cuenta". Nuestro sistema implementa la realidad bancaria bajo la cardinalidad 1:N:
	
	\begin{itemize}
		\item \textbf{El ID de Cliente (Expediente Base):} Representa a la persona física. Es un número generado aleatoriamente. Sirve como llave de acceso privada interna para el banco (\texttt{GestorBanco}).
		\item \textbf{La Cuenta Básica (El Medio Líquido):} Es el producto principal de captación. Su número (ej. rango 100000 - 900000) es público para que terceros realicen depósitos.
		\item \textbf{La Cuenta de Inversión (El Contrato Fijo):} Posee un número de folio completamente distinto (rango 500000+) porque legal y financieramente es un contrato a plazo fijo (28, 91 o 182 días), no una cuenta transaccional. Su separación permite que el dinero esté congelado y sujeto a cálculo de interés compuesto, sin mezclarse con el saldo disponible para compras diarias.
	\end{itemize}
	Esta normalización de datos asegura que el banco pueda ofrecer escalabilidad: un solo cliente puede tener tres tarjetas de crédito y dos inversiones bajo un mismo paraguas de identidad.
	
	\subsection{Evolución a Tipos Primitivos}
	A diferencia del primer acercamiento, todos los identificadores cambiaron de \texttt{String} a \texttt{int}. A nivel computacional, comparar cadenas (\texttt{String.equals()}) requiere analizar arreglos de caracteres, lo cual tiene un costo de tiempo de Big-O lineal donde la longitud importa. Usar enteros (\texttt{int}) reduce las comparaciones en memoria a una operación de CPU directa en tiempo constante, eficientando radicalmente la ejecución de las tablas \textit{Hash}.

	\section{Análisis de Estructuras de Datos (Collections)}
	El \textit{Java Collections Framework} proporciona la base para el almacenamiento en memoria. Se seleccionaron tres interfaces distintas resolviendo problemas específicos:
	
	\subsection{Búsqueda Transaccional: Uso de Map (HashMap)}
	Implementado en \texttt{GestorBanco} como \texttt{Map<Integer, Cliente> directorioClientes}. 
	\begin{itemize}
		\item \textbf{Justificación Algorítmica:} En un banco, la operación más repetida es buscar el expediente del cliente antes de cualquier retiro o depósito. Un arreglo convencional (\texttt{ArrayList}) forzaría una búsqueda lineal de complejidad de tiempo lineal. El \texttt{HashMap} \cite{oracle_hashmap} utiliza una función de dispersión matemática (\textit{hashing}) sobre el ID entero, encontrando al cliente instantáneamente incluso si el banco tuviera millones de usuarios.
	\end{itemize}
	
	\subsection{Integridad Financiera: Uso de Set (HashSet)}
	Implementado en la clase \texttt{Cliente} para almacenar productos: \texttt{Set<CuentaBasica> cuentas}.
	\begin{itemize}
		\item \textbf{Justificación Algorítmica:} La regla fundamental de bases de datos es evitar duplicidad de llaves primarias. Un \texttt{Set} por definición matemática no permite elementos duplicados. Si un fallo del sistema intentara asignarle al cliente un contrato previamente existente, la estructura lo rechaza silenciosamente sin necesidad de programar iteraciones manuales restrictivas.
	\end{itemize}
	
	\subsection{Auditoría y LIFO: Uso de LinkedList}
	Implementado en \texttt{CuentaBasica} y \texttt{TarjetaCredito} como \texttt{LinkedList<Movimiento> historial}.
	\begin{itemize}
		\item \textbf{Justificación Algorítmica:} Originalmente se planteó un \texttt{ArrayList}, pero un Estado de Cuenta debe mostrar siempre la transacción más reciente en la cima. Esto es el clásico modelo Pila (\textit{Stack}) o Comportamiento \textbf{LIFO} (Último en entrar, primero en salir). Usando \texttt{LinkedList} y su método nativo \texttt{addFirst()}, insertamos objetos en la cabecera rápidamente. Si usáramos una lista estática, Java tendría que desplazar todos los elementos subsecuentes en memoria, lo cual causaría un cuello de botella terrible con historiales largos.
	\end{itemize}

	\section{Diseño Orientado a Objetos: Archivos y Responsabilidades}
	El sistema cumple con el Principio de Responsabilidad Única (SRP). A continuación se analizan los componentes y las líneas de código críticas explicadas mediante sus propios comentarios técnicos:
	
	\subsection{1. Clase Principal: \texttt{SistemaBancario.java}}
	Actúa como el hilo conductor (Main) de la aplicación. Aloja el bucle \texttt{while} que mantiene activo al "Cajero" mediante la clase \texttt{Scanner}.
	\begin{itemize}
		\item \textbf{Lógica destacada (\texttt{sn.nextLine();}):} Tras un método como \texttt{sn.nextInt()}, el carácter invisible \textit{Enter} queda flotando en el buffer del teclado. El sistema ejecuta repetidamente este comando de limpieza de buffer como técnica de programación defensiva para evitar que el escáner lance excepciones o salte menús.
	\end{itemize}
	
	\subsection{2. Clase Controladora: \texttt{GestorBanco.kjava}}
	Es el cerebro operativo. Administra el directorio global y actúa de puente entre el mundo exterior (Main) y las entidades internas. 
	\begin{itemize}
		\item \textbf{Comentario Técnico del Código:} El gestor no manipula los saldos matemáticamente, delega esa responsabilidad (Encapsulamiento) al objeto \texttt{CuentaBasica}.
	\end{itemize}
	
	\subsection{3. La Entidad Contenedora: \texttt{Cliente.java}}
	Almacena los datos personales e instancia colecciones \texttt{HashSet} vacías en su constructor.
	\begin{itemize}
		\item \textbf{Lógica destacada (\texttt{return !cuentas.isEmpty();}):} Una validación elegante para los requisitos del banco. El cliente debe tener cuenta básica para abrir inversiones. La expresión evalúa si la colección no está vacía, retornando la verdad absoluta en una sola línea de código sin requerir condicionales voluminosas.
	\end{itemize}
	
	\subsection{4. Productos Activos: \texttt{CuentaBasica.java} y \texttt{TarjetaCredito.java}}
	Ambos extienden la lógica de retiros y depósitos, pero la tarjeta aplica validaciones de límite de endeudamiento (\texttt{deudaActual + monto <= limiteCredito}).
	\begin{itemize}
		\item \textbf{Comentario Técnico del Código:} \texttt{// Guardamos el movimiento al principio de la lista (LIFO)}. Empleado justo después de alterar la variable \texttt{saldo}, garantizando que cada cambio de variable se acompañe atómicamente de un registro inmutable en el historial.
	\end{itemize}
	
	\subsection{5. Producto a Plazo: \texttt{CuentaInversion.java}}
	Contiene fórmulas matemáticas y condicionales estrictas para asignar la tasa de interés anual con base en validaciones de plazo de días (28, 91, 182). Retorna montos flotantes calculando rendimiento por días divididos entre el año financiero (360 días).
	
	\subsection{6. La Auditoría: \texttt{Movimiento.java}}
	Es una clase inmutable. Sus variables solo se asignan por constructor.
	\begin{itemize}
		\item \textbf{Lógica destacada:} Hace uso de \texttt{LocalDateTime.now()} de \texttt{java.time}. Abandona el simple formato de fecha e incluye \textit{timestamps} completos (horas, minutos, segundos), esencial para reportes de auditoría pericial bancaria.
	\end{itemize}

	\section{Detección y Corrección de Errores de Lógica ("Bugs")}
	Durante el ciclo de desarrollo se detectaron errores semánticos graves que no detenían la compilación, pero destruían la lógica contable financiera. Estos fueron corregidos en el código As-Built:
	
	\begin{enumerate}
		\item \textbf{Vulnerabilidad de "Dinero Fantasma":} En los primeros \textit{builds}, el método \texttt{abrirCuentaInversion} simplemente instanciaba el objeto. Esto significaba que el cliente obtenía un instrumento financiero creados "de la nada", sin que su saldo principal disminuyera.
		\item \textbf{Solución Estructural implementada:} Se obligó al sistema a pedir el número de cuenta de fondeo. El método ahora intercepta el flujo, intenta ejecutar \texttt{cuentaOrigen.retirar(monto)}. Si este método devuelve \texttt{false} (sin saldo), el flujo de inversión aborta instantáneamente. Esta misma corrección se aplicó al pago de la Tarjeta de Crédito.
		\item \textbf{Memory Leaks por Referencias Huérfanas:} Cuando una inversión finalizaba y se pagaba al usuario, el objeto seguía vivo en la memoria del \texttt{Set}. Se implementó el comando \texttt{remove} para eliminar la referencia exacta, permitiendo que el \textit{Garbage Collector} de Java libere la RAM.
	\end{enumerate}

	\subsection{Análisis de la Evolución y Cambios Estructurales}
	Al comparar la Figura \ref{fig:uml_inicial} con la Figura \ref{fig:uml_final}, se evidencian ajustes críticos impulsados por necesidades técnicas reales descubiertas durante el desarrollo:
	
	\begin{itemize}
		\item \textbf{Absorción de la Interfaz Visual:} En el modelo inicial existía una clase \texttt{InterfazCajero}. Durante el desarrollo, se comprobó que esta capa añadía un nivel de indirección innecesario para un sistema CLI. Por ello, se fusionó su responsabilidad directamente en la clase \texttt{SistemaBancario} (Main), reduciendo el acoplamiento y centralizando el uso de la clase \texttt{Scanner}.
		\item \textbf{Corrección de la Multiplicidad del Crédito:} El diagrama original establecía una relación $0..1$ entre Cliente y Tarjeta de Crédito (limitando al usuario a poseer una sola tarjeta). En la versión final, esto se corrigió a $0..*$ (cero a muchas), alojadas dentro de un \texttt{Set<TarjetaCredito>}, reflejando la realidad comercial bancaria.
		\item \textbf{Trazabilidad de Historial Completo:} Originalmente, solo la Cuenta Básica generaba registros de \texttt{Movimiento}. En la implementación final, se trazó una nueva línea de agregación ($0..*$) desde \texttt{TarjetaCredito} hacia \texttt{Movimiento}, obligando al sistema a auditar también las compras a crédito y abonos de deuda.
		\item \textbf{Refactorización de Firmas y Retornos:} El diagrama final muestra cómo los métodos de \texttt{GestorBanco} mutaron. Por ejemplo, la creación de inversiones pasó de recibir solo un monto, a exigir un \texttt{cuentaOrigen: int}. Este cambio en el diseño fue la respuesta directa para parchar la vulnerabilidad de "dinero fantasma" explicada en la Sección 5.
	\end{itemize}

	\section{Conclusiones}
	El desarrollo del ``Sistema Bancario'' es un caso de estudio exitoso en la aplicación del \textit{Collections Framework} y los algoritmos de validación asimétrica. Se comprobó que una correcta abstracción en la Programación Orientada a Objetos permite simular normativas financieras estrictas del mundo real. La transición documentada desde el diagrama UML preliminar hasta el modelo As-Built final demuestra un proceso iterativo maduro. El uso orquestado de estructuras de almacenamiento complejas, como \texttt{HashMap} y listas doblemente enlazadas, otorga al sistema un nivel de solidez computacional listo para su evaluación académica.
